Part 2: The Foundation — Event Sourcing & CQRS
Chapter 2: Rethinking Data — Why State Is a Lie
2.1 The Concept
In traditional applications, you store state. A user's balance is $100. When they deposit, you UPDATE balance SET amount = amount + 50. The previous value is gone forever.

In event sourcing, you store events. You store "User deposited $50 at 3:14 PM" and "User withdrew $20 at 3:15 PM." The balance is calculated by replaying events.

The profound implication: Your database is not the source of truth. Your event log is. The database is a cache—a disposable projection of truth. This is not optional in fintech. It is regulatory.

2.2 The Theory: Why This Matters in Fintech
Scenario	Traditional Database	Event Sourcing
Bug that miscalculates balance	Data corrupted, need backups, possible manual reconciliation	Replay events with fix, data intact, audit trail preserved
Audit requirement	Complex logging, easy to miss, regulators suspicious	Every state change is a recorded event with metadata
Need to understand why	Only know final state, must reconstruct from logs	Full history of decisions with context
Transaction rollback	Complex, may leave artifacts, requires compensating transactions	Simply don't append the event—the event never existed
Regulatory inquiry	"Show me how this balance changed over time" — days of work	Single query: SELECT * FROM events WHERE stream_id = ?
The Invariant Rule: In event sourcing, you never delete or update. You only append. This is immutable truth. If a regulator asks "What happened to this account on March 15, 2024?" you can answer with cryptographic certainty.

2.5 The Elite Insight: Eventual Consistency Is a Feature, Not a Bug
Junior developers fear eventual consistency. Senior architects embrace it. Elite engineers design for it.

What it means: When a user deposits money, the read model (what they see in the UI) updates milliseconds later, not instantly.

Why this is acceptable:

The event store knows the truth. If the read model is wrong, it's temporary. The system is honest about uncertainty.

Your UI should indicate pending states. Show a spinner. Be honest with users. This builds trust.

Financial systems have delays anyway. Wire transfers take hours. ACH takes days. A 50ms delay is invisible.

You can offer strong consistency when needed. For critical reads, bypass the read model and query the event store directly.

The elite pattern: Use all four based on context:

elixir
def get_balance(account_id, consistency \\ :eventual) do
  case consistency do
    :eventual ->
      # Fast path: read model (50ms)
      ReadModel.get_balance(account_id)

    :strong ->
      # Slow path: event store (200ms, but guaranteed correct)
      account = Aggregate.load(account_id)
      account.balance

    :cached_with_fallback ->
      # Try cache, fallback to event store if stale
      case Cache.get(account_id) do
        {:ok, balance} -> balance
        {:stale, _} -> Aggregate.load(account_id).balance
        {:error, :not_found} -> Aggregate.load(account_id).balance
      end
  end
end

Chapter 3: The Aggregate — Your Consistency Guardian
3.1 The Concept
An aggregate is a consistency boundary. It ensures that business rules are enforced before any event is recorded.

Think of it as a bouncer at a club: The bouncer (aggregate) checks ID (validates command), ensures no one already inside (consistency), and then lets people in (emits events). The bouncer never leaves the door.

3.2 The Theory: Aggregate Design Principles
Principle 1: Aggregates Are Transactional Boundaries

If two operations must happen atomically (both succeed or both fail), they belong in the same aggregate.

Example: In a ledger, debiting account A and crediting account B must be atomic. This is why we need a Saga (process manager) to coordinate across aggregates.

Principle 2: Aggregates Should Be Small

Each aggregate instance should represent a single consistency unit. For a bank, each account is its own aggregate instance. This allows parallel processing—Account #12345 and Account #67890 can be handled simultaneously.

Principle 3: Aggregates Are Pure State Machines

An aggregate should have no side effects. It should not:

Send emails

Call external APIs

Write to databases

Publish to message queues

It only:

Receives commands

Validates them against current state

Produces events

Why? Because aggregates are replayed from events. If they had side effects, replaying would trigger them again.

3.5 The Elite Insight: Optimistic Concurrency with Retry Logic
When two commands hit the same aggregate simultaneously, only one can win.

How it works: Every event has a version number. When you append events, you specify the expected version. If another command already appended, your version will be wrong and the command fails.

This is intentional. The elite approach:

elixir
defmodule LedgerFlow.CommandDispatcher do
  use Retry

  def dispatch(command, opts \\ []) do
    retries = Keyword.get(opts, :retries, 3)
    delay_ms = Keyword.get(opts, :delay_ms, 100)

    retry with: exponential_backoff(delay_ms) |> take(retries) do
      case Commanded.dispatch(command, consistency: :strong) do
        {:ok, _} -> {:ok, :success}
        {:error, :stale_version} -> {:retry, :concurrent_modification}
        {:error, other} -> {:error, other}
      end
    after
      {:error, :max_retries_exceeded} ->
        Logger.error("Command failed after #{retries} retries: #{inspect(command)}")
        {:error, :concurrent_modification_timeout}
    end
  end
end
Never use pessimistic locking. It kills performance and creates deadlocks. Optimistic concurrency with retries is the fintech standard.

Chapter 4: The Event Store — Your Immutable Truth
4.1 The Concept
The event store is the source of truth. It's an append-only log of every state change that has ever occurred in your system.

Think of it like a bank statement: You don't erase transactions. You append new ones. The current balance is always a calculation from the transaction history.

4.2 The Theory: Why Postgres for Event Store
Many developers assume event sourcing requires specialized databases like EventStoreDB. For 99% of applications, Postgres is superior.

Consideration	Specialized Event Store	Postgres
Operations	New skill, new tool	Already know it
Transactions	Limited or none	Full ACID
Tooling	Limited	Mature ecosystem
Cost	Additional license	Usually already paid for
Performance	Excellent	Excellent for 99% of needs
Backup/Recovery	Vendor-specific	Standard tools (pg_dump, WAL)
The exception: If you need >10,000 events per second or global distribution, consider EventStoreDB or Kafka.

4.4 The Elite Insight: Upcasters for Zero-Downtime Evolution
The Problem: Your business evolves. Event structures change. You add fields, rename them, remove obsolete data.

The Naive Solution: Migrate all old events. This requires downtime and risks data corruption.

The Elite Solution: Upcasters.

An upcaster transforms old event versions to new versions at read time as events are replayed.

elixir
# Example: You renamed "amount" to "transaction_amount"

defmodule Upcasters.MoneyDeposited do
  use Commanded.Event.Upcaster
  @version 2  # Current version of this event

  def upcast(%{"amount" => amount} = event, metadata, %{version: 1}) do
    # Transform version 1 to version 2
    event
    |> Map.put("transaction_amount", amount)
    |> Map.delete("amount")
    |> Map.put("__version__", 2)
  end

  def upcast(event, _metadata, _version), do: event
end

# Later, you add a new field with a default
defmodule Upcasters.MoneyDeposited do
  # Version 3 adds "currency" field

  def upcast(event, _metadata, %{version: 2}) do
    event
    |> Map.put("currency", "USD")  # Default for old events
    |> Map.put("__version__", 3)
  end

  def upcast(event, _metadata, _version), do: event
end
The Rule: Never migrate events in the database. Always use upcasters. This allows zero-downtime evolution and keeps your database schema stable.

Chapter 5: The Read Model — Your Query Cache
5.1 The Concept
The read model is a denormalized view of your event data, optimized for queries.

Why it exists: Event stores are append-only logs. Querying them for "all accounts with balance > 1000" would require replaying every event for every account. That's slow.

The solution: Projectors listen to events and update specialized query tables.

5.2 The Theory: CQRS with Materialized Views
CQRS (Command Query Responsibility Segregation) means:

Commands (writes) go to aggregates → events

Queries (reads) go to read models

5.3 The Elite Insight: Read Model Strategies by Consistency Need
Consistency Need	Strategy	Example
Eventual	Async projector	Balance display, transaction list
Strong	Query event store	Audit reports, fraud checks
Cached	Redis with TTL	Exchange rates, fee calculations

elixir
defmodule LedgerFlow.BalanceReader do
  # Three-tier strategy for balance reads

  def get_balance(account_id, opts \\ []) do
    consistency = Keyword.get(opts, :consistency, :eventual)

    case consistency do
      :eventual ->
        # Fast path: 50ms, may be stale by milliseconds
        ReadModel.Balance.get(account_id)

      :strong ->
        # Slow path: 200ms, guaranteed correct
        account = AccountAggregate.load(account_id)
        account.balance

      :cached_with_validation ->
        # Medium path: 100ms, validates with event store if stale
        case Cache.get(account_id) do
          {:ok, balance, version} ->
            # Check if cache is behind
            current_version = EventStore.get_version(account_id)
            if version == current_version do
              balance
            else
              # Rebuild from events
              account = AccountAggregate.load(account_id)
              Cache.put(account_id, account.balance, account.version)
              account.balance
            end
          {:error, :not_found} ->
            account = AccountAggregate.load(account_id)
            Cache.put(account_id, account.balance, account.version)
            account.balance
        end
    end
  end
end

Chapter 6: Process Managers (Sagas) — Coordinating Aggregates
6.1 The Concept
A saga (or process manager) coordinates multiple aggregates to achieve a business goal that spans consistency boundaries.

Example: Transferring money from Account A to Account B requires:

Withdraw from A (if fails, stop)

Deposit to B (if fails, compensate by depositing back to A)

This is a saga. In fintech, sagas are how you implement distributed transactions without two-phase commit.

6.2 The Theory: Saga Patterns
There are two saga patterns:

Pattern	Description	Use Case
Choreography	Events trigger actions, no central coordinator	Simple, linear flows, low complexity
Orchestration	Central saga tracks progress, issues commands	Complex flows with error handling, audit requirements
For fintech, use orchestration. You need central visibility into transaction status and the ability to answer "What happened to transfer #12345?"

6.4 The Elite Insight: Idempotency & Dead Letter Queues
The critical property: All saga commands must be idempotent.

If a saga retries a command (because of network issues), it should have the same effect as the first attempt.

elixir
defmodule LedgerFlow.Sagas.TransferSaga do
  use Commanded.ProcessManagers.ProcessManager,
    name: "TransferSaga",
    router: LedgerFlow.Router

  defstruct [
    :transfer_id,
    :from_account,
    :to_account,
    :amount,
    :status,
    :withdrawal_completed?,
    :deposit_completed?,
    :retry_count
  ]

  # Handle command with idempotency check
  def execute(%__MODULE__{status: :withdrawing} = saga, %WithdrawMoney{} = cmd) do
    if saga.withdrawal_completed? do
      # Already did this step, skip
      []
    else
      # First time, execute
      [cmd]
    end
  end

  # Handle compensation with idempotency
  def compensate(%__MODULE__{status: :compensating} = saga, %DepositMoney{} = cmd) do
    if saga.compensation_completed? do
      []
    else
      [cmd]
    end
  end

  # Dead letter queue for failed commands
  def handle_failure(%__MODULE__{} = saga, command, error) do
    new_retry_count = saga.retry_count + 1

    if new_retry_count <= 5 do
      # Retry with backoff
      Process.send_after(self(), {:retry, command}, 1000 * new_retry_count)
      {:noreply, %{saga | retry_count: new_retry_count}}
    else
      # Move to dead letter queue for manual intervention
      DeadLetterQueue.push(%{
        saga: saga,
        command: command,
        error: error,
        timestamp: DateTime.utc_now()
      })

      # Alert on-call engineer
      Alert.send("Transfer saga failed: #{saga.transfer_id}", error)

      {:noreply, %{saga | status: :failed_manual_intervention_required}}
    end
  end
end
Why this matters: Without idempotency, a network hiccup could double-withdraw. Without dead letter queues, you could lose transactions. Sagas are how fintech systems maintain consistency across boundaries.

Chapter 7: Docker — Packaging for Immutability
7.1 The Concept
Docker packages your application and its environment into a single artifact that runs identically everywhere.

The senior insight: The same Docker image that passes tests on your Mac should run in production. No exceptions.

The elite insight: You need two Dockerfiles—one for development with hot reload, one for production with releases.

7.2 The Theory: Multi-Stage Builds for Production
A multi-stage build:

Builder stage: Has compilers, dependencies, builds the release

Runtime stage: Only contains the compiled release, no build tools

This keeps images small (50-100 MB instead of 1-2 GB) and removes the compiler from production (security critical).

defmodule LedgerFlow.Health do
  def liveness? do
    # Just check the process is running
    Process.whereis(LedgerFlowWeb.Endpoint) != nil
  end

  def readiness? do
    # Check all dependencies
    case Repo.query("SELECT 1") do
      {:ok, _} -> true
      {:error, _} -> false
    end and
    case AMQP.Connection.connected?() do
      true -> true
      false -> false
    end
  end

Chapter 9: Nginx — The Edge Bodyguard
9.1 The Concept
Nginx sits at the edge of your network, handling:

SSL termination (expensive crypto in C, not Erlang)

Static file serving (CSS, JS, images)

Rate limiting at the IP level

Request filtering

Gzip compression

DDoS mitigation

Why Nginx, not Phoenix: Phoenix is fast, but Nginx is written in C and mathematically superior at static file serving and SSL. Let it do what it's optimized for.

9.3 The Elite Insight: Rate Limiting at Multiple Layers
The most expensive resource in your system is a BEAM process. Nginx can block malicious IPs before they ever wake up an Erlang scheduler.

Elite approach: Three layers of rate limiting:

Nginx (IP-based): Blocks brute force attacks before they reach your app

Phoenix (user-based): Per-user quotas for authenticated users

Aggregate (business rules): Business logic limits (e.g., "no more than 5 deposits per hour")

nginx
# Layer 1: IP-based limits (Nginx)
limit_req_zone $binary_remote_addr zone=login:10m rate=10r/m;

# Layer 2: Token-based limits (Nginx with auth header)
limit_req_zone $http_authorization zone=authenticated:10m rate=1000r/m;
elixir
# Layer 3: Business logic limits (Phoenix plug)
defmodule LedgerFlow.Plugs.UserRateLimit do
  def init(opts), do: opts

  def call(conn, _opts) do
    user_id = conn.assigns.current_user.id

    case RateLimiter.check_rate("user:#{user_id}:deposit", 5, 3600) do
      :allow ->
        conn
      {:deny, :rate_limited} ->
        conn
        |> put_status(429)
        |> json(%{error: "Rate limit exceeded. Maximum 5 deposits per hour."})
        |> halt()
    end
  end
end
elixir
# Layer 4: Aggregate invariants (Commanded)
defmodule LedgerFlow.Account do
  def execute(%Account{} = account, %DepositMoney{amount: amount}) do

    |> Enum.filter(&(&1.timestamp > DateTime.add(DateTime.utc_now(), -3600)))

    if length(recent_deposits) >= 5 do
      {:error, :deposit_rate_limit_exceeded}
    else
      # ... process deposit
    end
  end
end

Chapter 10: HAProxy — The Traffic Controller
10.1 The Concept
HAProxy is a TCP/HTTP load balancer that distributes traffic across your Elixir nodes. Unlike Nginx, HAProxy specializes in connection management and health checking.

Why HAProxy over Nginx for load balancing: HAProxy is optimized for high-connection-count scenarios (like WebSockets) and provides better observability into connection distribution.

    mode http
    stats enable
    stats uri /
    stats refresh 10s
    stats auth admin:admin
    stats admin if TRUE

# ==================== FRONTEND (Where users connect) ====================

    bind *:9000
    mode http

    # Log format with timing
    option httplog
    capture request header Host len 32
    capture request header User-Agent len 64
    capture request header X-Forwarded-For len 32

    # Define backends based on path

# ==================== WEB SOCKET BACKEND ====================

Type	How It Works	Use Case
Active	HAProxy pings /health every N seconds	Primary health detection
Passive	HAProxy observes actual traffic failures	Catch intermittent issues

# Active: aggressive detection (2 seconds)

# Passive: conservative (traffic-based)

  def with_trace_id(trace_id, fun) do
    # Set trace ID in Logger metadata

Chapter 13: Load Testing with k6
13.1 The Concept
Load testing tells you where your system breaks. You gradually increase load until you find the breaking point, then you reinforce it.

The goal: Know your limits before your users find them.

13.2 The Theory: Load Test Patterns
Pattern	Description	When to Use
Smoke Test	1-5 users for 1 minute	Verify basics, CI/CD
Load Test	Expected peak load for 10 minutes	Validate capacity

15.5 The Elite Insight: Circuit Breakers with Three States
When a dependency is failing, your app should fail fast rather than waiting for timeouts.

elixir
defmodule LedgerFlow.CircuitBreaker do
  use Fuse, name: :postgres, opts: [
    {:timeout, 5000},      # 5 second timeout
    {:failures, 5},        # 5 failures before opening
    {:reset, 30_000},      # 30 seconds before retry
    {:max_failures, 10}    # After 10 failures, stay open longer
  ]

  # Three states:
  # - CLOSED: Normal operation, passing requests through
  # - OPEN: Failing fast, not calling dependency
  # - HALF-OPEN: Testing if dependency has recovered

  def call_database(query) do
    case Fuse.ask(:postgres) do
      :ok ->
        # Circuit is CLOSED, try the query
        try do
          result = Repo.query(query)
          Fuse.ok(:postgres)  # Success, reset failure count
          {:ok, result}
        rescue
          e ->
            Fuse.failure(:postgres)  # Record failure
            {:error, e}
        end

      :blown ->
        # Circuit is OPEN, fail fast
        Logger.warning("Circuit breaker open for Postgres, failing fast")
        {:error, :circuit_open}

      :half_open ->
        # Circuit is HALF-OPEN, try a single request
        try do
          result = Repo.query(query)
          Fuse.reset(:postgres)  # Success, close circuit
          {:ok, result}
        rescue
          e ->
            Fuse.failure(:postgres)  # Failure, reopen circuit
            {:error, e}
        end
    end
  end
end
Why: Without circuit breakers, when Postgres is slow, all your Elixir processes wait for timeouts, exhausting all resources. With circuit breakers, you fail immediately and preserve capacity for healthy requests.

          aws elbv2 modify-listener \
            --listener-arn ${{ secrets.LISTENER_ARN }} \
            --default-actions Type=forward,TargetGroupArn=${{ secrets.TG_$NEW }}

          # Drain old environment
          sleep 30
          aws ecs update-service \
            --cluster ledger-flow-prod \
            --service ledger-flow-$CURRENT \
            --desired-count 0
16.4 The Elite Insight: OIDC Authentication
Never store AWS keys in GitHub secrets. Use OIDC (OpenID Connect) for temporary, short-lived credentials.

json
// AWS IAM Role trust policy for OIDC
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::123456789012:oidc-provider/token.actions.githubusercontent.com"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
        },
        "StringLike": {
          "token.actions.githubusercontent.com:sub": "repo:your-org/ledger-flow:*"
        }
      }
    }
  ]
}
Why this matters: No long-lived secrets. If a developer's laptop is compromised, attackers can't steal AWS keys because they don't exist. The tokens expire in 60 minutes and are scoped to specific repositories.

Part 7: The Elite Final Test — Game Day
Chapter 17: The Game Day Exercise
17.1 The Concept
A Game Day is a planned disaster simulation. You deliberately break your system while observing how it responds.

The goal: Build confidence that your system self-heals and that your team knows how to respond.

17.2 The Elite Final Test Scenarios
Scenario	How to Simulate	Expected Behavior
RabbitMQ Outage	docker stop rabbitmq	Commands buffer or fail gracefully, system recovers when RabbitMQ restarts
Postgres Latency Spike	Toxiproxy latency injection	Circuit breakers trip, system returns 503 quickly
Elixir Node Failure	docker stop node1	HAProxy detects failure, traffic shifts, WebSocket connections failover
Network Partition	docker network disconnect	libcluster detects partition, Horde ensures single instance of processes
Database Failover	Force RDS failover	Application reconnects automatically, no manual intervention
AZ Outage	Stop instances in one AZ	ASG launches in other AZ, load balancer shifts traffic
Clock Skew	Set incorrect time on node	Distributed system handles skew without corruption
Memory Leak	Run soak test for 24 hours	Memory stabilizes, no OOM kills
17.3 The Complete Game Day Script
bash
#!/bin/bash
# /scripts/game_day.sh — Complete Disaster Simulation

set -e

# Colors
RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
BLUE='\033[0;34m'
NC='\033[0m'

# Configuration
LOAD_TEST_DURATION=300  # 5 minutes
LOAD_TEST_USERS=500
ENVIRONMENT=${ENVIRONMENT:-"staging"}
ALERT_WEBHOOK=${ALERT_WEBHOOK:-""}

echo -e "${BLUE}╔══════════════════════════════════════════════════════════════╗${NC}"
echo -e "${BLUE}║                    GAME DAY STARTING                         ║${NC}"
echo -e "${BLUE}║                    $(date)                                    ║${NC}"
echo -e "${BLUE}╚══════════════════════════════════════════════════════════════╝${NC}"

# Function to send alert
send_alert() {
    local message=$1
    echo -e "${YELLOW}[ALERT] $message${NC}"

    if [ -n "$ALERT_WEBHOOK" ]; then
        curl -s -X POST "$ALERT_WEBHOOK" \
            -H "Content-Type: application/json" \
            -d "{\"text\":\"[GAME DAY] $message\"}"
    fi
}

# Function to record metric
record_metric() {
    local name=$1
    local value=$2

    echo "$name: $value" >> /tmp/game_day_metrics.log
}

# Function to check system health
check_health() {
    local service=$1
    local max_retries=${2:-10}
    local retry=0

    while [ $retry -lt $max_retries ]; do
        if curl -s -f "http://localhost:4000/health" > /dev/null; then
            return 0
        fi
        sleep 2
        retry=$((retry+1))
    done
    return 1
}

# Function to get metric
get_metric() {
    local metric=$1
    curl -s "http://localhost:9090/api/v1/query?query=$metric" | jq -r '.data.result[0].value[1] // 0'
}

# ==================== PREPARATION ====================
echo -e "\n${BLUE}📊 Phase 0: Preparation${NC}"
echo "================================"

# Record baseline metrics
echo "Recording baseline metrics..."
BASELINE_LATENCY=$(get_metric "histogram_quantile(0.99, rate(phoenix_request_duration_microseconds_bucket[5m]))")
BASELINE_ERRORS=$(get_metric "rate(phoenix_http_errors_total[5m])")
BASELINE_RABBITMQ=$(get_metric "rabbitmq_queue_depth")
record_metric "baseline_latency" "$BASELINE_LATENCY"
record_metric "baseline_errors" "$BASELINE_ERRORS"
record_metric "baseline_rabbitmq" "$BASELINE_RABBITMQ"

# Start load test
echo "Starting load test with $LOAD_TEST_USERS users..."
k6 run --vus $LOAD_TEST_USERS --duration ${LOAD_TEST_DURATION}s /load_tests/k6/spike_test.js &
K6_PID=$!

# Wait for load to stabilize
echo "Waiting for load to stabilize (30 seconds)..."
sleep 30
check_health "system" || {
    echo -e "${RED}✗ System not healthy before tests!${NC}"
    exit 1
}

# ==================== TEST 1: RabbitMQ Outage ====================
echo -e "\n${BLUE}🐇 Phase 1: RabbitMQ Outage${NC}"
echo "================================"

send_alert "TEST 1: Killing RabbitMQ"

# Record pre-test metrics
PRE_RABBITMQ_QUEUE=$(get_metric "rabbitmq_queue_depth")
echo "Pre-RabbitMQ queue depth: $PRE_RABBITMQ_QUEUE"

# Kill RabbitMQ
docker stop ledger_flow_rabbitmq_1 2>/dev/null || true
echo "RabbitMQ killed at $(date)"

# Monitor for 30 seconds
sleep 30

# Check system health during outage
if check_health "system" 5; then
    echo -e "${GREEN}✓ System remains healthy during RabbitMQ outage${NC}"
    record_metric "rabbitmq_outage_system_healthy" "true"
else
    echo -e "${RED}✗ System unhealthy during RabbitMQ outage!${NC}"
    record_metric "rabbitmq_outage_system_healthy" "false"
fi

# Check if commands are being buffered or failing gracefully
POST_RABBITMQ_QUEUE=$(get_metric "rabbitmq_queue_depth")
echo "Post-RabbitMQ queue depth: $POST_RABBITMQ_QUEUE"

# Restart RabbitMQ
echo "Restarting RabbitMQ..."
docker start ledger_flow_rabbitmq_1 2>/dev/null || true

# Wait for recovery
sleep 30

# Verify recovery
if check_health "system" 10; then
    echo -e "${GREEN}✓ System recovered after RabbitMQ restart${NC}"
    record_metric "rabbitmq_outage_recovered" "true"
else
    echo -e "${RED}✗ System did not recover after RabbitMQ restart!${NC}"
    record_metric "rabbitmq_outage_recovered" "false"
fi

# ==================== TEST 2: Postgres Latency ====================
echo -e "\n${BLUE}🐘 Phase 2: Postgres Latency Spike${NC}"
echo "================================"

send_alert "TEST 2: Injecting 2s Postgres latency"

# Inject latency
curl -s -X POST http://localhost:8474/proxies/postgres_proxy/toxicities \
    -H "Content-Type: application/json" \
    -d '{"type":"latency","attributes":{"latency":2000,"jitter":100}}' > /dev/null

echo "Postgres latency injected at $(date)"

# Wait for circuit breakers to trip
sleep 15

# Check circuit breaker status
CIRCUIT_STATUS=$(curl -s http://localhost:4000/metrics | grep circuit_postgres || echo "circuit_postgres 0")
echo "Circuit breaker status: $CIRCUIT_STATUS"

if echo "$CIRCUIT_STATUS" | grep -q "circuit_postgres 1"; then
    echo -e "${GREEN}✓ Circuit breaker tripped as expected${NC}"
    record_metric "postgres_latency_circuit_open" "true"
else
    echo -e "${RED}✗ Circuit breaker did not trip!${NC}"
    record_metric "postgres_latency_circuit_open" "false"
fi

# Check system response time
LATENCY_DURING=$(get_metric "histogram_quantile(0.99, rate(phoenix_request_duration_microseconds_bucket[1m]))")
echo "P99 latency during injection: ${LATENCY_DURING}ms"

# Remove latency
sleep 30
curl -s -X DELETE http://localhost:8474/proxies/postgres_proxy/toxicities/latency > /dev/null

# Wait for recovery
sleep 30

# Check circuit breaker reset
CIRCUIT_AFTER=$(curl -s http://localhost:4000/metrics | grep circuit_postgres || echo "circuit_postgres 0")
if echo "$CIRCUIT_AFTER" | grep -q "circuit_postgres 0"; then
    echo -e "${GREEN}✓ Circuit breaker reset after recovery${NC}"
    record_metric "postgres_latency_circuit_closed" "true"
fi

# ==================== TEST 3: Elixir Node Failure ====================
echo -e "\n${BLUE}💀 Phase 3: Elixir Node Failure${NC}"
echo "================================"

send_alert "TEST 3: Killing Elixir node 1"

# Get initial connection distribution
CONNECTIONS_BEFORE=$(curl -s http://localhost:1936/ | grep -c "node[0-9]")
echo "Active nodes before: $CONNECTIONS_BEFORE"

# Kill node
docker stop ledger_flow_elixir_node1_1 2>/dev/null || true
echo "Elixir node killed at $(date)"

# Wait for HAProxy to detect failure
sleep 10

# Check HAProxy status
HAPROXY_STATUS=$(curl -s http://localhost:1936/ | grep -i "node1" || echo "Node1 down")
echo "HAProxy status: $HAPROXY_STATUS"

# Verify traffic shifted
CONNECTIONS_AFTER=$(curl -s http://localhost:1936/ | grep -c "node[0-9]")
echo "Active nodes after: $CONNECTIONS_AFTER"

# Check if system is still serving traffic
if check_health "system" 5; then
    echo -e "${GREEN}✓ System continues serving traffic with one node down${NC}"
    record_metric "node_failure_system_healthy" "true"
else
    echo -e "${RED}✗ System failed with one node down!${NC}"
    record_metric "node_failure_system_healthy" "false"
fi

# Restart node
sleep 30
docker start ledger_flow_elixir_node1_1 2>/dev/null || true
sleep 15

# Verify node rejoins cluster
NODES_ACTIVE=$(docker exec ledger_flow_elixir_node1_1 bin/ledger_flow eval "Node.list()" | wc -l)
if [ "$NODES_ACTIVE" -ge 2 ]; then
    echo -e "${GREEN}✓ Node rejoined cluster${NC}"
    record_metric "node_failure_rejoined" "true"
fi

# ==================== TEST 4: Network Partition ====================
echo -e "\n${BLUE}🌐 Phase 4: Network Partition${NC}"
echo "================================"

send_alert "TEST 4: Creating network partition"

# Create partition
docker network disconnect backend ledger_flow_elixir_node2_1 2>/dev/null || true
echo "Network partition created at $(date)"

# Wait for libcluster to detect
sleep 10

# Check cluster view
CLUSTER_STATUS=$(docker exec ledger_flow_elixir_node1_1 bin/ledger_flow eval "Node.list()" || echo "[]")
echo "Cluster view from node1: $CLUSTER_STATUS"

# Verify no split-brain (critical)
# Check if any processes are duplicated across partitions
DUPLICATE_PROCESSES=$(docker exec ledger_flow_elixir_node1_1 bin/ledger_flow eval \
    "Horde.Cluster.members(LedgerFlow.ProcessRegistry) |> length()" || echo "0")
echo "Unique processes: $DUPLICATE_PROCESSES"

# Reconnect
sleep 30
docker network connect backend ledger_flow_elixir_node2_1 2>/dev/null || true
sleep 15

# Verify cluster reconnected
CLUSTER_AFTER=$(docker exec ledger_flow_elixir_node1_1 bin/ledger_flow eval "Node.list()" | wc -l)
if [ "$CLUSTER_AFTER" -ge 2 ]; then
    echo -e "${GREEN}✓ Cluster reconnected after partition${NC}"
    record_metric "network_partition_recovered" "true"
fi

# ==================== TEST 5: Traffic Spike ====================
echo -e "\n${BLUE}⚡ Phase 5: Traffic Spike${NC}"
echo "================================"

send_alert "TEST 5: 10x traffic spike"

# Record pre-spike metrics
PRE_SPIKE_LATENCY=$(get_metric "histogram_quantile(0.99, rate(phoenix_request_duration_microseconds_bucket[1m]))")
echo "Pre-spike P99 latency: ${PRE_SPIKE_LATENCY}ms"

# Run spike test in parallel
k6 run --vus 2000 --duration 60s /load_tests/k6/spike_test.js &
SPIKE_PID=$!

# Monitor during spike
sleep 30
SPIKE_LATENCY=$(get_metric "histogram_quantile(0.99, rate(phoenix_request_duration_microseconds_bucket[30s]))")
echo "During spike P99 latency: ${SPIKE_LATENCY}ms"

# Wait for spike to complete
wait $SPIKE_PID

# Wait for recovery
sleep 30
POST_SPIKE_LATENCY=$(get_metric "histogram_quantile(0.99, rate(phoenix_request_duration_microseconds_bucket[1m]))")
echo "Post-spike P99 latency: ${POST_SPIKE_LATENCY}ms"

if (( $(echo "$POST_SPIKE_LATENCY < 500" | bc -l) )); then
    echo -e "${GREEN}✓ System recovered after spike${NC}"
    record_metric "spike_recovered" "true"
fi

# ==================== FINAL VERIFICATION ====================
echo -e "\n${BLUE}🏆 Phase 6: Final Verification${NC}"
echo "================================"

# Stop load test
kill $K6_PID 2>/dev/null || true

# Final health check
sleep 30
if check_health "system" 20; then
    echo -e "${GREEN}✓ System fully healthy after all tests${NC}"
    record_metric "final_health" "true"
else
    echo -e "${RED}✗ System unhealthy after tests!${NC}"
    record_metric "final_health" "false"
fi

# Compare metrics

FINAL_ERRORS=$(get_metric "rate(phoenix_http_errors_total[5m])")

echo ""
echo -e "${BLUE}╔══════════════════════════════════════════════════════════════╗${NC}"
echo -e "${BLUE}║                    GAME DAY RESULTS                          ║${NC}"
